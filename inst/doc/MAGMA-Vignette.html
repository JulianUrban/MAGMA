MAGMA - MAny Group MAtching

Introduction

The MAGMA-package offers nearest neighbor matching for two to four groups. It also includes the possibility to match data of a 2x2 Design. Besides this possibility to match many groups, MAGMA introduces a framework to evaluate post matching balance. This vignette is a tutorial of MAGMA. It shows the main MAGMA function on an included simulated data set. Overall, this vignette includes a two-group, a three-group, and a 2x2-design which is also approximately equivalent to a four-group example. This three examples are conducted in a standard matching process as well as with exact matching. Parts of this tutorial are also included on the respective help pages of the functions. For details of the algorithm and the theoretical background of MAGMA, see Feuchter et al.Â (2022).

This simulated data set contains 800 cases and 14 variables. It is available after loading the MAGMA package. Associations between the variables were modeled to increase usability and comprehensibility of this tutorial. Any associations are, therefore, not interpretable. Two-Group Example

In this fictional two-group example the variable gifted_support is the independent (i.e., the treatment) variable. We are interested in how the receiving of giftedness support affects the college GPA. However, other variables, such as intelligence are associated with both, giftedness support and the college GPA. Therefore, we aim to reduce the effect of these covariates through the matching. For this example, the covariates are the high school GPA, intelligence, motivation, sex, and whether at least one parent has an academic background. Note that MAGMA can only consider metric and binary variables for balance estimation.

covariates_gifted <- c("GPA_school", "IQ_score", "Motivation", "parents_academic", "sex")

After this definition we can estimate the initial unbalance over all cases in our data set. Here, the function initial_unbalnce estimates the four balance criteria pre-matching. Therefore, it is necessary to define the data set (MAGMA_sim_data), the grouping variable (gifted_support), and all binary and metric covariates of interest (covariates_gifted). This result in a vector with the length four - the four balance criteria.

unbalance_gifted <- initial_unbalance(data = MAGMA_sim_data,
                                      group = "gifted_support",
                                      covariates = covariates_gifted)
#> [1] "mean g was computed using random effects meta-analysis with metafor."
unbalance_gifted
#>           Pillai's Trace d-ratio Mean g adj. d-ratio
#> Unbalance           0.13     0.4   0.31          0.4

As may be expected from the descriptive statistics, the cases that received giftedness support differed at least on some variables substantially from the cases that did not receive giftedness support. In the next step, we compute the matching. This process is based on the respective propensity score, namely ps_gifted. We first address standard matching and deal after that with exact matching.

Standard Matching

For standard nearest neighbor matching we use the function MAGMA. This function has four arguments. The first specifies the data set (MAGMA_sim_data), the second the name of the grouping variable as a character ("gifted_support"), and the third the name of the distance variable/propensity score as a character ("ps_gifted"). The fourth argument is optional. Since MAGMA matching has some computational load it includes the possibility of parallel computation. Per default, only one core is used. If parallel computation is desired, a larger than one integer can be defined for the cores argument. If the specified value exceeds the number of available cores, MAGMA sets this argument to the maximum available cores.

Applying this function results in a data set, that augments the original data set for three variables. The first, weight, indicates whether this case was matched or not. The second, step, names the iterative step in which a case was matched. More specifically, the cases with the smallest distance are matched first and, thus, receive 1 in the step variable. The matched cases with the second smallest distance receive the 2 and so on. The third variable, distance specifies this distance of the matched cases.

MAGMA_sim_data_gifted <- MAGMA(Data = MAGMA_sim_data,
                                group = "gifted_support",
                                dist = "ps_gifted",
                                cores = 2)

Balance estimation

MAGMA matches cases iteratively until all cases of the samllest group are matched. Thus, it is necessary to find the optimal sample size after matching. To do so, Balance_MAGMA estimates the four balance criteria for each possible sample size. Note that MAGMA uses a lower limit of at least n = 20 per group. To apply the function you need at least to specify your data set (MAGMA_sim_data_gifted), the name of the grouping variable as a character ("gifted_support"), and your covariates as a character vector (covariates_gifted). Moreover, Balance_MAGMA needs the name of the variable, that indicates the iterative step in that a case was matched. Since the MAGMA function names this variable "step" this name is the default of the step argument. If you want to use Balance_MAGMA on a data set with various MAGMA-matchings or to evaluate balance of another matching algorithm, you need to define the respective name as a character.

Balance_MAGMA returns a list length four. In the case of a univariate grouping variable (compare 2x2 matching), this list contains three numeric vectors and one list length two. The three vectors form Pillaiâs Trace, mean g, and adjusted d-ratio over the iteratively increased sample size. The list length two includes one numeric vector, d-ratio, and one matrix. The matrix specifies all pairwise dâs for all covariates. Both, d-ratio and pairwise dâs, are also estimated teratively over the increased sample size. The example below estimates the Balance for the matching and shows the balance criteria and pairwise effects for a group sample size of n = 100.

Balance_gifted <- Balance_MAGMA(data = MAGMA_sim_data_gifted,
                                group = "gifted_support",
                                covariates = covariates_gifted,
                                step = "step") #Not necessary to define here

Balance plotting

With this information regarding the balance with respect to sample size, the desired sample size can be selected. Which sample size is âoptimalâ does strongly depend on the specific research aim. As example, sample size requirements of some statistical procedures may narrow the selection, or the need to eliminate a specific pairwise effect. We, therefore, want to encourage to adapt the selection of optimal sample size to the specific research aim.

A first indicator on such an optimal model is the trend of the balance criteria across the iteratively increased sample size. The function Plot_MAGMA visualize this trend. The one mandatory argument, Balance, is specifying the object of the balance estimation results. Note that this object needs to be the result of a Balance_MAGMA function. Per default, Plot_MAGMA creates plots for all four balance criteria. However, the criterion argument enables the plotting of specific balance criteria. Note that running this code leads to a warning. This warning is a result of the specified minimum sample size for balance estimation and can be ignored.

The trend of this plots strongly depends on your data. Nonetheless, they can be a good indicator, where breakpoints in the sample size-balance associations may be. Plot_MAGMA uses fixed limits for the y-axis.This ensures comparability of this plots and prevents misinterpretation due to a restricted y-axis range. Both, d-ratio and adjusted d-ratio, vary between 0 and 1, wherefore those form the limits for these criteria. Although Pillaiâs Trace has theoretically the same range, we set the limits to 0 and 0.5. Larger values are unlikely and indicate a bad balance. A stricter limitation could restrict a comprehensive trend evaluation. For mean g we set the limits to 0 and 1. The lower limit 0 corresponds to the theoretical lower limit. The rationality for the upper limit is comparable to the one of Pillaiâs Trace.

Plot_MAGMA(Balance = Balance_gifted,
           criterion = c("Pillai", "d_ratio", "mean_g", "Adj_d_ratio")) #Could be omitted

Balance table

Besides the trend over sample size the absolutely best values for each criterion are of interest. The function Table_MAGMA extracts these values and returns them in an APA table in an extra file. More specific it returns a 4x5 table. In each of the four rows one balance criterion has its optimal value. The five columns represent the sample size per group, where the respective criterion is optimal, and the four balance criteria with this sample size per group. Besides creating a file with an APA table, the table is printed to the console, too.

Similar to Plot_MAGMA the first argument Balance of Table-MAGMA specifies the object of the balance estimation results. The second argument filename indicates the desired name of the extra file with the APA table. This argument is a character.

Table_MAGMA(Balance = Balance_gifted,
            filename = "Balance_gifted.docx")

Exact Matching

In comparison to standard matching, only cases sharing the same value on a defined variable can be matched in exact matching. In this example, the variable for exact matching is enrichment. This means, that cases that participated in afternoon enrichment can only be matched with other cases that participated, and vice versa. Reasons to select exact matching instead of standard matching may be, for example, a nested data structure. Besides this preselection of possible matches the process of matching stays the same. The main differene is that we need to us MAGMA_exact instead of MAGMA to match the data. This function has a further argument, where the name of the exact variable needs to be defined as a character ("enrichment"). Note that the initial unbalance is the same as for standard matching.

MAGMA_sim_data_gifted_exact <- MAGMA_exact(Data = MAGMA_sim_data,
                                           group = "gifted_support",
                                           dist = "ps_gifted",
                                           exact = "enrichment",

Three-Group Example

In this fictional three-group example the variable teacher_ability_rating is the independent (i.e., the treatment) variable. We are interested in how the teacher rated ability ranging from below average to average and above average affects the college GPA. While the process of matching and balance evaluation is the same as in the two-group example, the grouping variable (teacher_ability_rating) and consequently the distance variable/propensity score changes (ps_tar). Moreover, we slightly change the covariates by substituting sex through gifted_support.

covariates_tar <- c("GPA_school", "IQ_score", "Motivation", "parents_academic", "gifted_support")

Standard Matching

Due to this changes in grouping variable and covariates, we need to estimate the initial unbalance for teacher rated ability. Note that due to having three groups the estimation of mean g slightly changes. However, this does only affect the backend computation, while the code and the display of results are not affected. For more details, see Feuchter et al.Â (2022).

unbalance_tar <- initial_unbalance(data = MAGMA_sim_data,
                                   group = "teacher_ability_rating",
                                   covariates = covariates_tar)
#> [1] "mean g was computed using robust variance meta-analysis with robumeta."
unbalance_tar
#>           Pillai's Trace d-ratio Mean g adj. d-ratio
#> Unbalance           0.23    0.47   0.35         0.47

2x2/four-group Example

In this fictional 2x2-design example the variables gifted_support and enrichment are the independent (i.e., the treatment) variables. We are interested in how the receiving of giftedness support and the participation in afternoon enrichment affects the college GPA. The process of matching and balance evaluation is the same as in the former examples. The grouping variable consists of two variables, wherefore we use a character vector as input for the groupung variable (c("gifted_support", "enrichment")). Consequently, we also adapt the distance variable/propensity score (ps_2x2). Moreover, we use the same covariates as in the two-group example. Note that this 2x2 matching is nearly equivalent to a four group matching using a multinominal variable indicating the four cells of the 2x2 group design. The multinominal variable for this example is support_enrichment. Major distinctions betwwen such a multinominal four-group matching and a 2x2 matching occur only for balance estimation. This differences are described more thoroughly below.

covariates_2x2 <- c("GPA_school", "IQ_score", "Motivation", "parents_academic", "sex")

Standard Matching

Due to this changes in grouping variable and covariates, we need to estimate the initial unbalance for the two independent variables gifted_support and enrichment. The result of the initial_unbalance function slightly changes in this two-factorial design. Pillaiâs Trace is estimated separately for the two main effects and the interaction. As can be seen, d-ratio, mean g, and adjusted d-ratio are the same for a 2x2 or a four-group matching. Changes only occur for Pillaiâs Trace.

After this check of initial unbalance we can conduct the matching. As mentioned, we need to adapt the grouping variable and the distance variable/propensity score to this example. Moreover, the conducting of a 2x2/four-group matching has some computational load. If a RAM threshold is exceeded, MAGMA computes quasi-systematic matching. The rationality of this is described elsewhere. Again, this only affects the computational backend and does not affect the user application. Note that in the case of a multifactorial matching, MAGMA returns a fourth variable. This fourth variable is a multinominal version of the two independent variables (group_long).
